Scraping TabNet

Extração e tratamento de dados de produção ambulatorial do SUS

DATASUS · Playwright · PostgreSQL

Apresentação técnica — IESB

Contexto: o que é o TabNet?

  • Sistema do DATASUS (Ministério da Saúde) para tabulação de dados públicos de saúde.
  • Disponibiliza estatísticas de mortalidade, morbidade, produção hospitalar e ambulatorial.
  • Interface web antiga (CGI), formulários HTML, sem API pública.
  • Dados de interesse: Produção Ambulatorial do SUS por município e subgrupo de procedimento.
Problema: dados ricos, mas só acessíveis por formulário web → necessidade de scraping.

Stack

  • Playwright — automação de navegador (Chromium headless)
  • BeautifulSoup + lxml — parsing do HTML de resposta
  • psycopg2 — cliente PostgreSQL
  • uv — gerenciamento de dependências e ambiente

Como funciona o scraping

  1. Usuário informa período inicial e final (MM/AAAA).
  2. Playwright abre o Chromium e navega até o formulário do TabNet.
  3. Configura campos fixos: Linha = Município, Coluna = Subgrupo_proced., Formato = prn (CSV ;).
  4. Para cada mês × {Qtd.aprovada, Valor_aprovado} (50 queries):
    • Seleciona o conteúdo e o período
    • Submete o formulário, captura o popup
    • Extrai a tabela do <pre> de resposta
  5. Grava cada lote no output.csv incrementalmente.
  6. Delay de 10s entre requisições (boas práticas).

Dados brutos (output.csv)

Formato wide: subgrupos como colunas, uma linha por (período, conteúdo, município).

Período;Conteúdo;Município;0101 Ações...;0102 Vigilância...;...;Total
Jan/2024;Qtd.aprovada;110001 ALTA FLORESTA;139;165;...;48352
Jan/2024;Valor_aprovado;110001 ALTA FLORESTA;-;-;...;102441,38
Jan/2024;Qtd.aprovada;Total;...;...;...
Jan/2024;Qtd.aprovada;000000 Ignorado;-;-;...;-
  • ~252.000 linhas brutas
  • 52 colunas de subgrupos (incluindo "Total")
  • Inclui linhas de "Total" e "Ignorado" — descartadas no tratamento

Tratamento de dados (load_db.py)

  1. Extração de códigos: regex pega o número no início do nome do subgrupo
    "0101 Ações de promoção..." → 0101
  2. Filtro: remove linhas com município contendo "Total" ou "ignorado".
  3. Pivot: cada combinação (período, município) vira uma única linha. Os dois conteúdos (Qtd.aprovada / Valor_aprovado) se transformam em pares de colunas: 
    qtd_0101, val_0101, qtd_0102, val_0102, ..., qtd_0803, val_0803
  4. Schema dinâmico: a tabela é recriada com colunas baseadas nos subgrupos detectados no CSV.
  5. Upsert: INSERT ... ON CONFLICT (periodo, municipio) DO UPDATE permite re-execução idempotente.

Estrutura final

CREATE TABLE producao_ambulatorial (
    periodo     VARCHAR(10) NOT NULL,
    municipio   TEXT        NOT NULL,
    qtd_0101 TEXT, val_0101 TEXT,
    qtd_0102 TEXT, val_0102 TEXT,
    ...
    qtd_0803 TEXT, val_0803 TEXT,
    PRIMARY KEY (periodo, municipio)
);

Cada linha agora cruza diretamente quantidade e valor por subgrupo — pronto para análise dimensional.

Conclusão

  • Reprodutível: CLI parametrizada, dependências travadas em uv.lock.
  • Resiliente: erros por requisição não interrompem a coleta; gravação incremental no CSV.
  • Idempotente: upsert na carga permite re-execução sem duplicar.
  • Flexível: schema gerado a partir dos dados, adapta-se a novos subgrupos.
